🏢 SOC là gì?

Security Operations Center (SOC) là trung tâm giám sát an ninh mạng, nơi các chuyên gia phân tích log, phát hiện mối đe dọa, điều tra sự cố và ứng phó sự cố (IR — Incident Response) theo thời gian thực.

💾 SIEM vs XDR

SIEM (Security Information and Event Management): Thu thập, tổng hợp, phân tích log từ nhiều nguồn (endpoint, firewall, IDS...) nhằm phát hiện hành vi bất thường và tạo cảnh báo.

XDR (Extended Detection and Response): Thế hệ kế tiếp của SIEM, tích hợp detection và response trên endpoint, network, server trong một nền tảng thống nhất.

🛠️ Các thành phần trong Lab

  • Wazuh Server — Nền tảng SIEM/XDR mã nguồn mở: Manager (thu thập log), Indexer (lưu trữ dữ liệu), Dashboard (trực quan hóa)
  • Wazuh Agent — Agent chạy trên endpoint, thu thập log và chuyển về Wazuh Server
  • pfSense — Firewall ảo hóa, cung cấp log firewall cho Wazuh
  • Suricata — IDS/IPS: giám sát lưu lượng mạng, phát hiện xâm nhập, có chế độ passive (IDS) và active (IPS)
  • Sysmon — Giám sát endpoint Windows, capture chi tiết system events
  • VirusTotal — Threat intelligence: API lookup để enrich cảnh báo với dữ liệu uy tín
  • FIM (File Integrity Monitoring) — Giám sát thay đổi file/thư mục real-time
  • Hydra — Công cụ brute-force (chỉ dùng trong môi trường lab cách ly)

2. Tích hợp Suricata IDS/IPS
Cài đặt Suricata trên Windows · Npcap · Detection rules · Log forwarding
▼

📖 Lý thuyết

Suricata là một IDS/IPS engine mã nguồn mở, có thể hoạt động ở hai chế độ:

  • IDS (Intrusion Detection System): Phát hiện xâm nhập thụ động — giám sát lưu lượng, sinh alerts nhưng không block.
  • IPS (Intrusion Prevention System): Phòng ngừa xâm nhập chủ động — phát hiện và block threats theo rules.

Suricata sử dụng các detection rules để nhận diện mẫu tấn công trong network traffic. Rules được cập nhật thường xuyên từ các nguồn như ET Open (Emerging Threats).

🔧 Các bước thực hiện

1
Cài đặt Suricata trên Windows
Tải và cài đặt Suricata. Cài đặt Npcap (thư viện packet capture cho Windows) làm yêu cầu bắt buộc.
2
Cấu hình network interface
Chỉ định network interface để Suricata giám sát (thường là card mạng kết nối internet).
3
Cấu hình detection rules
Tải và kích hoạt các rule sets (ET Open). Kiểm tra cấu hình suricata.yaml.
4
Tích hợp logs với Wazuh
Cấu hình Suricata EVE JSON log output. Cấu hình Wazuh Agent đọc và forward Suricata logs về Wazuh Server để tập trung hóa giám sát.
  • Cài đặt Suricata và Npcap trên Windows Thực hành
  • Cấu hình detection rules và EVE JSON logging Cấu hình
  • Tích hợp Suricata logs vào Wazuh Dashboard Thực hành
  • Kiểm tra IDS alerts trên Wazuh Dashboard Kiểm tra

3. Tích hợp pfSense Firewall
Deploy pfSense VM · Remote logging · Custom decoders & rules
▼

📖 Lý thuyết

pfSense là một firewall mã nguồn mở dựa trên FreeBSD, chạy được trên phần cứng thường hoặc máy ảo. Nó cung cấp khả năng:

  • Stateful packet filtering
  • VPN (IPSec, OpenVPN)
  • Traffic shaping
  • Remote syslog logging — gửi log về Wazuh

pfSense ghi log tất cả traffic được allow và deny, giúp SOC phân tích ai đang cố kết nối và bị block bởi firewall.

🔧 Các bước thực hiện

1
Deploy pfSense trong VMware
Tải pfSense ISO, tạo máy ảo mới, cấu hình network adapters (WAN + LAN).
2
Cấu hình Remote Logging
Trên pfSense Web UI → Status → System Logs → Settings → Remote. Điền địa chỉ Wazuh Server để gửi syslog.
3
Tạo Custom Decoders
Viết Wazuh decoder để parse pfSense log format. Decoder trích xuất các trường: source IP, destination IP, action (allow/deny), protocol, port.
4
Viết Custom Rules
Tạo Wazuh rules để phát hiện: authentication failures từ pfSense, blocked connections, port scans.
  • Deploy pfSense trong VMware và cấu hình mạng Thực hành
  • Bật remote syslog từ pfSense sang Wazuh Cấu hình
  • Viết custom decoder để parse pfSense logs Code
  • Viết Wazuh rules phát hiện firewall events Code

API key · Wazuh Integration · Real-time lookup · Alert enrichment

📖 Lý thuyết

VirusTotal là dịch vụ phân tích mối nguy (threat intelligence) miễn phí. Nó tổng hợp kết quả từ hàng chục antivirus engines và threat intelligence feeds.

Wazuh tích hợp VirusTotal API để khi phát hiện file hash hoặc IP đáng ngờ, hệ thống sẽ tự động tra cứu VirusTotal và enrich alert với:

  • Detection rate (bao nhiêu AV engines phát hiện là malware)

3
Cấu hình Wazuh Agent giám sát thư mục
Trong ossec.conf trên agent, thêm thư mục cần giám sát với real-time monitoring. Bật VirusTotal lookup cho syscheck.
4
Xác minh alert enrichment
Tạo file mới trong thư mục giám sát, kiểm tra Wazuh Dashboard để thấy alert được enrich với VirusTotal data.
  • Đăng ký và lấy VirusTotal API key Thực hành

5. File Integrity Monitoring (FIM)
Cấu hình syscheck · Real-time monitoring · Xác minh alerts
▼

📖 Lý thuyết

Wazuh FIM (File Integrity Monitoring) giám sát các thư mục và file quan trọng trên endpoint. Khi có thay đổi (create, modify, delete), Wazuh tạo alert ngay lập tức.

Điều này rất quan trọng để phát hiện:

  • Temporary file từ malware
  • Thay đổi file hệ thống bởi attacker
  • File mới được tạo bởi attacker
  • Cấu hình bị sửa đổi (registry, config files)

🔧 Các bước thực hiện

1
Cấu hình syscheck trong ossec.conf
Thêm thẻ <syscheck> vào ossec.conf trên agent. Liệt kê các thư mục cần giám sát (VD: C:\Windows\System32, C:\Users).
2
Bật real-time monitoring
Thêm thuộc tính realtime="yes" cho các thư mục. Bật recursion="yes" để giám sát cả thư mục con.
3
Xác minh hoạt động
Tạo file mới → kiểm tra alert "File added". Sửa file → kiểm tra alert "File modified". Xóa file → kiểm tra alert "File deleted".
<syscheck> <directories check_all="yes" realtime="yes">C:\Users\Documents</directories> <directories check_all="yes" whodata="yes">C:\Windows\System32</directories> </syscheck>
  • Cấu hình syscheck trong ossec.conf với thư mục cần giám sát Cấu hình
  • Bật real-time monitoring và recursion Cấu hình
  • Xác minh: tạo → alert "added", sửa → alert "modified", xóa → alert "deleted" Kiểm tra

📖 Lý thuyết — Sysmon

Sysmon (System Monitor) của Microsoft Sysinternals cung cấp log chi tiết hơn Event Viewer thôngường. Nó ghi:

  • Process creation/termination (Event ID 1)
  • Network connections (Event ID 3)
  • File creation (Event ID 11)
  • Registry changes (Event ID 12-14)
  • Drive access (Event ID 9)
  • WMI events (Event ID 19-21)

Sysmon là công cụ không thể thiếu trong DFIR (Digital Forensics & Incident Response) vì nó cung cấp bức tranh toàn cảnh về hoạt động trên endpoint.

🔧 Các bước thực hiện

1
Tìm hiểu Windows Event Logs
Mở Event Viewer trên Windows. Thực hành đọc các Event ID quan trọng: 4625 (failed login), 4672 (admin login), 6008 (shutdown).
2
Cài đặt Sysmon
Tải Sysmon từ Microsoft. Cài đặt với config file (SwiftOnSecurity config là phổ biến nhất cho lab).
3
Cấu hình Wazuh Agent thu thập Sysmon logs
Cấu hình Wazuh Agent đọc Sysmon event channel và forward logs về Wazuh Server.
4
Viết Wazuh rules phát hiện hành vi đáng ngờ
Tạo rules phát hiện: suspicious process (powershell-encoded-command), lateral movement, credential access.
  • Đọc và phân tích các Event ID quan trọng trong Event Viewer Lý thuyết

7. Brute Force Attack — Simulation, Detection & Defense
Hydra · Event ID 4625 · Wazuh rules · Biện pháp phòng thủ
▼
⚠️
Cảnh báo bảo mật: Chỉ thực hiện mô phỏng tấn công trong môi trường lab cách ly (các VM mô tả ở trên) hoặc trên hệ thống bạn sở hữu/được ủy quyền kiểm tra. Không bao giờ chạy brute-force trên hệ thống của bên thứ ba hoặc production.

📖 Lý thuyết — Brute Force Attack

Brute Force Attack là kỹ thuật tấn công dò password bằng cách thử liên tục tất cả các tổ hợp ký tự có thể. Trong SSH, attacker kết nối liên tục và thử hàng nghìn password mỗi phút.

Biến thể:

  • Simple brute force: Thử tất cả tổ hợp
  • Dictionary attack: Thử từ danh sách password phổ biến (hiệu quả hơn)
  • Credential stuffing: Dùng password đã leak từ data breach khác

🔧 Các bước thực hiện

1
Mô phỏng SSH brute force bằng Hydra
Trên Kali Linux, chạy Hydra để tấn công SSH service đang chạy trong lab. Hydra sẽ thử hàng nghìn password.
2
Phát hiện qua Windows Event Log (Event ID 4625)
Kiểm tra Event Viewer → Security → Event ID 4625 (An account failed to log on). Xem Source Network Address để xác định attacker IP.
3
Phát hiện qua Wazuh Dashboard
Xem alerts trên Wazuh Dashboard. Kiểm tra correlation rules đã được cấu hình để highlight authentication failures.
4
Áp dụng biện pháp phòng thủ
Triển khai: strong password policy, MFA, account lockout threshold, Wazuh active response (block attacker IP bằng firewall).
# Ví dụ lệnh Hydra (chỉ dùng trong lab cách ly!) hydra -l admin -P /usr/share/wordlists/rockyou.txt ssh://TARGET_IP -V

🛡️ Biện pháp phòng thủ

  • Xác minh Wazuh phát hiện brute force attack qua alerts Kiểm tra

📊 Các chỉ số đánh giá

  • Accuracy: Tổng số dự đoán đúng / Tổng số mẫu
  • Precision: TP / (TP + FP) — Trong số các mẫu dự đoán là attack, bao nhiêu đúng?
  • Recall (Sensitivity): TP / (TP + FN) — Trong số các attack thực sự, phát hiện được bao nhiêu?
  • F1-Score: 2 × (Precision × Recall) / (Precision + Recall) — Trung bình điều hòa
💡
Trong cybersecurity, Recall cho attack classes quan trọng hơn Precision. Bỏ sót một cuộc tấn công thực sự (FN) nguy hiểm hơn nhiều so với false alarm (FP).

🗂️ Access Control List (ACL) là gì?

ACL (Access Control List) là danh sách kiểm soát truy cập trên thiết bị mạng (Router, Switch) dùng để lọc lưu lượng mạng — cho phép (permit) hoặc từ chối (deny) packet dựa trên điều kiện định sẵn.

ACL hoạt động như một "người gác cổng" tại interface của thiết bị mạng. Mỗi packet đi qua interface đều được kiểm tra against ACL rules theo thứ tự từ trên xuống.

📂 Phân loại ACL

LoạiSố hiệuLọc theoVị trí đặtĐặc điểm
Standard ACL1-99, 1300-1999Chỉ Source IPGần destination nhấtĐơn giản, ít chi tiết
Extended ACL100-199, 2000-2699Source, Dest IP, Protocol, PortGần source nhấtChi tiết, linh hoạt hơn
Named ACLĐặt tênStandard hoặc ExtendedTùy loạiDễ quản lý, đặt tên mô tả

⚙️ Quy tắc xử lý của ACL

┌─────────────────────────────────────────────────┐ │ PACKET ĐẾN INTERFACE có ACL │ │ │ │ │ ┌───────────────┴───────────────┐ │ │ │ Kiểm tra ACL rule #1 │ │ │ │ ┌───────────┴───────────┐ │ │ │ │ │ MATCH? │ │ │ │ │ ├──────┴──────────────┤ │ │ │ │ │ YES NO │ │ │ │ │ ▼ ▼ │ │ │ │ DENY → DROP Tiếp tục │ │ │ │ │ kiểm tra │ │ │ │ rule #2... │ │ │ │ │ │ │ └────────────────────────────┘ │ │ KHÔNG MATCH rule nào │ │ │ │ │ ▼ │ │ IMPLICIT DENY ALL │ │ (Tự động từ chối mọi packet │ │ nếu không match rule nào) │ └─────────────────────────────────────────────────┘

🔄 Inbound vs Outbound ACL

  • Inbound ACL: Packet được kiểm tra TRƯỚC KHI định tuyến. Nếu deny → packet bị drop ngay, không tốn tài nguyên routing.
  • Outbound ACL: Packet được kiểm tra SAU KHI đã định tuyến, ngay trước khi rời interface.

Best practice: Đặt Extended ACL gần source (để drop sớm), Standard ACL gần destination (vì chỉ có source IP).

🌐 Wildcard Mask

Wildcard mask xác định phạm vi IP được match. Cách tính: 255.255.255.255 - subnet_mask = wildcard_mask

Wildcard MaskÝ nghĩaVí dụ
0.0.0.0Chính xác 1 IP192.168.1.1 0.0.0.0 = chỉ IP đó
0.0.0.255Tất cả IP trong subnet /24192.168.1.0 0.0.0.255 = 192.168.1.0/24
0.0.255.255Tất cả IP trong /16192.168.0.0 0.0.255.255 = 192.168.0.0/16
0.255.255.255Tất cả IP trong /810.0.0.0 0.255.255.255 = 10.0.0.0/8
255.255.255.255Mọi IP (any)any hoặc 0.0.0.0 255.255.255.255

📖 ACL Keywords đặc biệt

  • any — Khớp với mọi địa chỉ IP (tương đương 0.0.0.0 255.255.255.255)
  • host — Chỉ một IP cụ thể (tương đương với wildcard 0.0.0.0)
  • eq — Equal: khớp port cụ thể (VD: eq 80)
  • neq — Not equal: khớp port khác port chỉ định
  • gt — Greater than: port lớn hơn
  • lt — Less than: port nhỏ hơn
  • range — Khoảng port (VD: range 20 100)
